**מקטע רביעי: הרחבת שאילתת חיפוש**

בהמשך הקובץ מופיעה דוגמה מעשית יותר:

```python
query = "car"
most_similar = model.most_similar(query, topn=8)
expanded_terms = [w for w, score in most_similar]

print(f"Original query: {query}")
print("Expanded terms:", expanded_terms)
```

כאן אנחנו מתחילים לראות כיצד Word2Vec יכול להשתלב במערכת אמיתית.

המשתנה:

query = "car"

מייצג שאילתת חיפוש של משתמש. במקום לחפש רק את המילה car, אנחנו מבקשים מהמודל למצוא מילים קרובות במשמעות:

```python
most_similar = model.most_similar(query, topn=8) 
```

התוצאה היא רשימה של מילים שקשורות ל-car, למשל מילים מעולם הרכב, התחבורה או כלי הרכב.

לאחר מכן הקוד יוצר רשימה נקייה של המילים בלבד:

```python
expanded_terms = [w for w, score in most_similar]
```

כאן יש שימוש ב-List Comprehension של Python. הפונקציה most_similar מחזירה זוגות של מילה וציון דמיון. למשל:

("vehicle", 0.78)

אבל לצורך הרחבת שאילתה אנחנו רוצים כרגע רק את המילים, ולכן לוקחים את w ומתעלמים מה-score.

המשמעות ההנדסית היא ברורה: אם משתמש חיפש car, מערכת חיפוש יכולה להוסיף מאחורי הקלעים מילים קרובות, וכך להחזיר תוצאות רחבות יותר. זה נקרא Query Expansion. היתרון הוא שיפור ה-Recall, כלומר הגדלת הסיכוי למצוא מסמכים רלוונטיים גם אם הם לא השתמשו בדיוק באותה מילה שהמשתמש הקליד.

החיסרון הוא שצריך להיזהר. הרחבת שאילתה לא מבוקרת עלולה להוסיף מילים קרובות מדי מבחינה סטטיסטית אבל לא מדויקות מספיק לצורך העסקי. לכן במערכות אמיתיות משלבים בדרך כלל סף דמיון, בדיקות איכות, ולעיתים גם חוקים עסקיים.

**מקטע חמישי: התאמת פנייה לנושא שירות**

המקטע הבא מדגים שימוש מעשי נוסף:

```python
faq_topics = ["billing", "refund", "password", "shipping", "account"]

user_issue = "payment"

similarities = []
for topic in faq_topics:
    if topic in model and user_issue in model:
        score = model.similarity(user_issue, topic)
        similarities.append((topic, score))

best_topic = max(similarities, key=lambda x: x[1])

print("User issue:", user_issue)
print("Best matching FAQ topic:", best_topic)
```

כאן המטרה היא לקחת פנייה פשוטה של משתמש ולמצוא לאיזה נושא FAQ היא הכי קרובה.

הרשימה:

```python
faq_topics = ["billing", "refund", "password", "shipping", "account"]
```

מייצגת נושאים אפשריים במערכת תמיכה.

המשתנה:

user_issue = "payment"

מייצג את הבעיה שהמשתמש כתב, בצורה מאוד פשוטה ומצומצמת.

הלולאה עוברת על כל נושא אפשרי:

```python
for topic in faq_topics:
```

ובתוך הלולאה יש בדיקת בטיחות:

```python
if topic in model and user_issue in model:
```

הבדיקה הזו חשובה מאוד. מודל Word2Vec מכיר רק מילים שהיו באוצר המילים שלו בזמן האימון. אם ננסה לחשב דמיון למילה שאינה קיימת במודל, נקבל שגיאה. לכן הקוד בודק קודם שגם topic וגם user_issue קיימים במודל.

לאחר מכן מחושב דמיון סמנטי:

```python
score = model.similarity(user_issue, topic)
```

הפונקציה similarity מחזירה ציון שמבטא עד כמה שתי מילים קרובות זו לזו במרחב הווקטורי. ככל שהציון גבוה יותר, כך המילים קרובות יותר מבחינת ההקשרים שבהם הופיעו.

השורה:

```python
similarities.append((topic, score))
```

שומרת את הנושא ואת ציון הדמיון שלו.

בסוף בוחרים את הנושא עם הציון הגבוה ביותר:

```python
best_topic = max(similarities, key=lambda x: x[1])
```

המשמעות של

```python
key=lambda x: x[1]
```

היא: בחר את האיבר שהערך השני שלו, כלומר ציון הדמיון, הוא הגבוה ביותר.

**רגע לפני שממשיכים: מהו Tensor?**

כאשר עובדים עם מודלים של Deep Learning, רואים הרבה פעמים את המונח Tensor.

בפשטות, Tensor הוא מבנה נתונים מספרי. אפשר לחשוב עליו כהרחבה של רשימה או טבלה.

ברמה הפשוטה ביותר:

[1, 2, 3]

זו רשימה חד ממדית של מספרים. בעולם המתמטי אפשר לקרוא לה וקטור.

דוגמה דו ממדית:

```python
[
    [1, 2, 3],
    [4, 5, 6]
]
```

זו טבלה של מספרים. בעולם המתמטי אפשר לקרוא לה מטריצה.

אבל במודלים של Deep Learning אנחנו צריכים לעיתים לעבוד עם מבנים עמוקים יותר, למשל כמה משפטים, שכל אחד מהם מכיל כמה Tokens, וכל Token מיוצג על ידי וקטור ארוך של מספרים. כאן נכנס המונח Tensor.

לדוגמה, כאשר BERT מקבל משפט, הוא לא עובד עם הטקסט עצמו. המשפט עובר Tokenization, וכל Token הופך למספר. לאחר מכן המודל מייצר לכל Token ייצוג וקטורי.

אם נעבד משפט אחד שיש בו 10 Tokens, וכל Token מיוצג על ידי 768 מספרים, נקבל מבנה בגודל:

1 x 10 x 768

המשמעות היא:

1 משפט אחד

10 עשרה Tokens

768 מספרים שמייצגים כל Token

המבנה הזה נקרא Tensor.

לכן, כאשר בקוד מופיע:

```python
inputs = tokenizer(sentence, return_tensors="pt")
```

הפרמטר:

return_tensors="pt"

אומר ל-Tokenizer להחזיר את הנתונים בפורמט של PyTorch Tensor, כלומר בפורמט מספרי שמתאים ישירות לעבודה עם מודל Deep Learning שרץ באמצעות PyTorch.

חשוב להבין: Tensor אינו מושג מיוחד ל-BERT בלבד. זהו מבנה בסיסי בעולם של Deep Learning. תמונה, טקסט, קול או וידאו, כולם הופכים בסופו של דבר ל-Tensors כדי שמודל יוכל לבצע עליהם חישובים.

Tensor הוא מבנה נתונים מספרי רב ממדי, שבו מודלים של Deep Learning משתמשים כדי לייצג קלטים, פלטים וייצוגים פנימיים. במקרה של BERT, המשפט הופך ל Tensor שמכיל מזהים מספריים של Tokens, ולאחר מכן ל Tensor עשיר יותר שבו כל Token מיוצג על ידי וקטור של מאות מספרים.

חשוב לא להתבלבל בין **Vector** לבין **Tensor**. כאשר אומרים שמילה הופכת לווקטור, הכוונה היא שכל Token מקבל רשימת מספרים שמייצגת אותו. כאשר עובדים עם מודל כמו BERT, המודל אינו מקבל וקטור אחד בלבד, אלא משפט שלם שמורכב מרצף Tokens. לכן כל הווקטורים של כל ה-Tokens נארזים יחד במבנה מספרי גדול יותר שנקרא Tensor. במילים אחרות: הווקטור מייצג Token אחד, וה-Tensor הוא המבנה שמכיל את כל הווקטורים שהמודל מעבד.

**מקטע חמישי: הדגמת Lemmatization**

```python
def demo_lemmatization(words: list[str]) -> None:
    """Show lemmatization using POS-aware normalization."""
    lemmatizer = WordNetLemmatizer()
    print("=== Lemmatization (WordNetLemmatizer) ===")
    for word in words:
        noun_lemma = lemmatizer.lemmatize(word)
        verb_lemma = lemmatizer.lemmatize(word, pos="v")
        print(f"{word:>12} -> noun:{noun_lemma:<10} verb:{verb_lemma}")
    print()
```

כדי שהכלי ינתח את המילה נכון, אפשר לעזור לו ולציין מה התפקיד שלה במשפט. זה נעשה באמצעות הפרמטר pos, שהוא קיצור של Part of Speech. כאשר אנחנו כותבים pos="v", אנחנו בעצם אומרים ל-lemmatize: התייחס למילה הזאת כפועל.

למה זה חשוב?

כי אותה מילה יכולה להתנהג אחרת לפי חלק הדיבר שלה. לדוגמה, המילה ate היא פועל בעבר. אם המערכת יודעת שמדובר בפועל, היא יכולה להחזיר את צורת הבסיס:

```bash

```

אבל אם היא מתייחסת אליה כברירת מחדל כשם עצם, ייתכן שהיא לא תדע לשנות אותה.

זו נקודה חשובה מאוד ב-NLP: כדי לבצע ניתוח לשוני טוב, לא תמיד מספיק לראות את המילה עצמה. לעיתים צריך לדעת איזה תפקיד היא ממלאת במשפט.

ומתקבל פלט שמציג את אותה רשימת מילים בשתי גישות שונות: פעם אחת באמצעות Stemming, ופעם אחת באמצעות Lemmatization.

```bash
Original words:
running, studies, better, wolves, flying, ate, cars, happiest

=== Stemming (PorterStemmer) ===
running -> run
studies -> studi
better -> better
wolves -> wolv
flying -> fli
ate -> ate
cars -> car
happiest -> happiest

=== Lemmatization (WordNetLemmatizer) ===
running -> noun:running verb:run
studies -> noun:study verb:study
better -> noun:better verb:better
wolves -> noun:wolf verb:wolves
flying -> noun:flying verb:fly
ate -> noun:ate verb:eat
cars -> noun:car verb:cars
happiest -> noun:happiest verb:happiest

Note:
- Stemming is faster but can produce non-words (e.g. 'studi').
- Lemmatization is more linguistic and usually cleaner.
```

ב-Stemming, האלגוריתם חותך את המילים בצורה טכנית ומהירה. לכן studies הופכת ל-studi, ו-wolves הופכת ל-wolv. אלה אינן מילים תקניות באנגלית, אבל מבחינת האלגוריתם הן מספיקות כדי לצמצם מילים דומות לצורה בסיסית משותפת.

לעומת זאת, ב-Lemmatization, הכלי מנסה להגיע לצורת בסיס לשונית תקינה. לכן wolves הופכת ל-wolf, ו-ate הופכת ל-eat כאשר מציינים שמדובר בפועל. זו דוגמה טובה לכך ש-Lemmatization מבין יותר מהתהליך הלשוני, ולא רק חותך סיומות.

שימו לב במיוחד לשורה:

```bash
running -> noun:running    verb:run
```

כאשר המילה running מנותחת כשם עצם, היא נשארת running. אבל כאשר אנו אומרים לכלי להתייחס אליה כפועל, היא הופכת ל-run. זו בדיוק הסיבה לכך שהמידע על התפקיד של המילה במשפט חשוב כל כך.

גם השורה:

```bash
ate -> noun:ate        verb:eat
```

מדגימה את אותו רעיון. אם הכלי מתייחס ל-ate כשם עצם, הוא לא משנה אותה. אבל כאשר מציינים שמדובר בפועל, הוא יודע להחזיר אותה לצורת הבסיס eat.

זו נקודה חשובה מאוד: Lemmatization אינו רק ניקוי טכני של טקסט. הוא תלוי בהבנה לשונית בסיסית של תפקיד המילה. לכן במערכות NLP מתקדמות, לעיתים משלבים לפניו גם ניתוח תחבירי או זיהוי Part of Speech, כדי לדעת האם מילה מסוימת מתפקדת כשם עצם, פועל או שם תואר.

הפלט גם מראה שלשתי הגישות יש מגבלות. לדוגמה, better ו-happiest לא משתנות כאן בצורה משמעותית. זה מזכיר לנו שגם כלים לשוניים מוכנים אינם קסם. איכות התוצאה תלויה בכלי, בהגדרות, במידע שנמסר לו, ובשפה שבה עובדים.

מבחינה לימודית, תוצאת ההרצה חשובה לא פחות מהקוד עצמו. היא מאפשרת לראות בפועל מה ההבדל בין גישה מהירה וגסה לבין גישה מדויקת יותר שמנסה לשמור על משמעות לשונית.

השם nlp הוא שם מקובל מאוד בדוגמאות של spaCy. הוא מייצג את צינור העיבוד כולו. כלומר, המשתנה הזה אינו רק מודל NER, אלא pipeline שלם שיודע לקבל טקסט ולהחזיר אובייקט עשיר עם ניתוחים לשוניים.

השורה הבאה:

```python
doc = nlp(SAMPLE_TEXT)
```

מעבירה את הטקסט דרך ה-pipeline.

כאן מתבצע בפועל עיבוד ה-NLP. הטקסט עובר Tokenization, ניתוחים פנימיים, ובין היתר גם זיהוי ישויות.

התוצאה נשמרת במשתנה:

doc

האובייקט doc הוא לא סתם מחרוזת. זהו אובייקט של spaCy שמכיל את הטקסט המקורי, את ה-Tokens, ואת המידע שהמודל חילץ ממנו.

**מקטע חמישי: מעבר על הישויות שזוהו**

לאחר שהטקסט עבר עיבוד, הקוד מדפיס כותרת:

```python
print("Built-in NER (en_core_web_sm)\n")
```

ואז עובר על הישויות:

```python
for ent in doc.ents:
```

המאפיין:

```python
doc.ents
```

מכיל את כל הישויות שהמודל זיהה בטקסט.

כל ent בלולאה הוא ישות אחת. למשל, ייתכן ש-ent.text יהיה:

Apple Inc.

ו-ent.label_ יהיה:

ORG

כלומר, המודל זיהה את הביטוי Apple Inc. כארגון.

בתוך הלולאה מופיעה השורה:

```python
explain = spacy.explain(ent.label_) or ""
```

הפונקציה:

```python
spacy.explain(ent.label_)
```

מקבלת קוד תווית כמו ORG, PERSON או DATE, ומחזירה הסבר קריא יותר.

לדוגמה:

ORG

יכול לקבל הסבר כמו:

```python
Companies, agencies, institutions, etc.
```

החלק:

```python
or ""
```

אומר שאם spaCy לא מצליח למצוא הסבר לתווית, נשתמש במחרוזת ריקה במקום לקבל ערך ריק שעלול להפריע בהדפסה.

לבסוף מופיעה שורת ההדפסה:

```python
print(f"  {ent.text!r:40} | {ent.label_:<10} | {explain}")
```

זו שורת עיצוב פלט.

החלק:

```python

```

מדפיס את טקסט הישות ברוחב של 40 תווים. הסימן !r אומר להדפיס את הייצוג הרשמי של המחרוזת, בדרך כלל עם גרשיים. זה עוזר לראות בבירור את גבולות הטקסט שזוהה.

החלק:

```python
{ent.label_:<10}
```

מדפיס את תווית הישות ומיישר אותה לשמאל בתוך רוחב של 10 תווים.

בסוף מודפס ההסבר האנושי של התווית.

התוצאה היא טבלה טקסטואלית פשוטה שמציגה:

הטקסט שזוהה | סוג הישות | הסבר

עם זאת, הפלט גם מדגים נקודה חשובה מאוד: מודל NLP אינו מושלם.

דווקא הטעות הזאת חשובה מאוד מבחינה לימודית. היא מזכירה לנו שמודל מוכן כמו en_core_web_sm יכול להיות שימושי מאוד, אבל אסור להתייחס אליו כאל מנגנון מושלם לזיהוי כל סוגי המידע. אם בונים מערכת אמיתית להסתרת מידע רגיש, למשל הסתרת טלפונים, כתובות אימייל או מספרי זהות, לא מספיק להסתמך רק על מודל NER כללי. במקרים כאלה צריך לשלב כלים נוספים, כמו ביטויים רגולריים, חוקים עסקיים, מודלים ייעודיים ל-PII, ובדיקות איכות על נתוני אמת.

לכן, תוצאת ההרצה מלמדת שני דברים במקביל. מצד אחד, היא מראה את הכוח של NER: היכולת לחלץ מתוך טקסט חופשי שמות של ארגונים, אנשים, מקומות ותאריכים. מצד שני, היא מראה את המגבלה: המודל עלול לטעות, במיוחד כאשר הוא נתקל בפרטים טכניים כמו מספרי טלפון או כתובות אימייל.
