# מהו MCP באמת

## מה MCP מנסה לפתור: בעיית N×M

בפרק 1 הצגנו את בעיית N×M: כשכל מודל מתחבר לכל כלי בנפרד, מספר האינטגרציות גדל בצורה שלא ניתן לתחזק. שלושה מודלים וארבעה כלים שווים שתים עשרה אינטגרציות שונות. כל אחת עם הלוגיקה שלה, הפורמט שלה, ומנגנון האימות שלה.

הבעיה הזו לא ייחודית לעולם ה-AI. היא מוכרת בהנדסת תוכנה מזה עשורים. הפתרון הקלאסי לה הוא תקן משותף: במקום שכל צד ידבר עם כל צד אחר ישירות, כולם מדברים דרך שפה אחת מוסכמת.

MCP הוא התקן הזה עבור חיבור מודלים לכלים ומשאבים.

הרעיון פשוט: מודל שיודע לדבר MCP יכול לעבוד עם כל כלי שחושף את עצמו דרך MCP, בלי שום קוד אינטגרציה נוסף. כלי שנכתב פעם אחת לפי התקן יכול לשרת כל מודל שמדבר אותו. הבעיה עוברת מ-N×M ל-N+M. זה לא רק חיסכון בקוד. זה שינוי ארכיטקטוני שמאפשר לבנות מערכות שניתן לתחזק, להרחיב, ולאבטח בצורה עקבית.

## MCP כתקן פתוח: לא מוצר, לא API פרטי

MCP פותח על ידי Anthropic ב-2024, אבל נקודה אחת חשובה להבין מיד: הוא לא מוצר של Anthropic. הוא תקן פתוח שהועבר לניהול של Linux Foundation, אותה גוף שמנהל תשתיות קוד פתוח קריטיות כמו Linux, Kubernetes ו-Node.js.

המשמעות של זה מעשית לחלוטין.

כל חברה, כל מתכנת, וכל ארגון יכולים לממש MCP בלי לשלם רישיון, בלי תלות בספק יחיד, ובלי חשש שהתקן ישתנה לפי האינטרסים של חברה אחת. זה מה שמבדיל תקן פתוח מ-API פרטי.

API פרטי הוא ממשק שחברה מגדירה לשימוש השירות שלה. אם החברה משנה את ה-API, כל מי שמשתמש בו צריך להתאים את עצמו. אם החברה סוגרת את השירות, האינטגרציה מפסיקה לעבוד. אם החברה מחליטה לגבות תשלום, אין ברירה.

MCP עובד אחרת. ההגדרה של הפרוטוקול פתוחה לעיון, השינויים בו מתבצעים בשקיפות עם מעורבות של הקהילה, וכל מימוש שעומד בתקן עובד עם כל מימוש אחר. זה בדיוק המודל שאפשר לבנות עליו תשתית ארגונית לטווח ארוך.



## השחקנים המרכזיים במבנה: Host, Client, Server, Model

כדי להבין איך MCP עובד, צריך להכיר את ארבעת השחקנים שמרכיבים כל מערכת מבוססת MCP. כל אחד מהם אחראי על חלק ברור, ואין חפיפה בין האחריויות.

**Host** הוא הסביבה שבה הכל מתרחש. זה יכול להיות סביבת הפיתוח שלך כמו Cursor או VS Code, אפליקציה שבנית שמשלבת מודל, או כל סביבה אחרת שמאפשרת למודל לפעול. ה-Host הוא זה שמחזיק את חוויית המשתמש, מנהל את ההרשאות, ומחליט אילו שרתי MCP מחוברים ופעילים. מנקודת המבט של אבטחה, ה-Host הוא נקודת האמון של המשתמש.

**Client** הוא שכבת התקשורת שחיה בתוך ה-Host. הוא זה שמדבר עם שרתי MCP בשם ה-Host. כשמודל רוצה לקרוא לכלי, ה-Client הוא זה שמנהל את התקשורת מול השרת המתאים, מחזיר את התוצאה, ומוודא שהפרוטוקול מכובד משני הצדדים.

**Server** הוא ספק היכולות. הוא חושף כלים, משאבים ותבניות דרך ממשק MCP מוגדר. שרת MCP יכול להיות שירות שמאפשר גישה למסד נתונים, כלי שמבצע חיפוש במסמכים, או כל יכולת אחרת שרוצים לחשוף למודל. כל שרת אחראי על תחום אחד בלבד, ולא יותר.

**Model** הוא הרכיב שמקבל הקשר ומחזיר החלטה או פעולה. חשוב להבין: המודל הוא לא חלק רשמי מפרוטוקול MCP. הוא צרכן ההקשר שה-Host מספק לו. הוא לא מתקשר ישירות עם שרתי MCP, אלא עובד מול ה-Host שמתווך עבורו.

ארבעת השחקנים האלה עובדים יחד בכל קריאה לכלי. המשתמש פועל מול ה-Host. ה-Host מעביר את הבקשה למודל. המודל מחליט שצריך לקרוא לכלי. ה-Client מעביר את הקריאה לשרת המתאים. השרת מחזיר תוצאה. ה-Client מעביר אותה בחזרה למודל. המודל ממשיך משם.

כל שלב ברצף הזה הוא נקודת שליטה שאפשר לבדוק, לאכוף ולתעד.

## מה MCP מסדיר ומה הוא לא מסדיר

אחת הטעויות הנפוצות כשמתחילים לעבוד עם MCP היא לצפות שהפרוטוקול יפתור בעיות שהוא לא נועד לפתור. חשוב להבין את הגבולות בדיוק כמו שחשוב להבין את היכולות.

**מה MCP מסדיר:** MCP מגדיר את שפת התקשורת בין Host לבין Server.

הוא קובע:

- איך מכריזים על יכולות

- איך בונים קריאה לכלי

- איך מחזירים תוצאה

- איך מדווחים על שגיאות

- איך מנהלים את מחזור החיים של חיבור.

אלה כל ההיבטים שהפרוטוקול מגדיר.

**מה MCP לא מסדיר:**

MCP לא מגדיר מי רשאי לקרוא לאיזה כלי. הוא לא קובע מה קורה כשכלי נכשל. הוא לא מגדיר איך שומרים לוגים, איך מנהלים Rate Limiting, איך מבצעים אימות משתמשים, או איך מחליטים אם פעולה מסוכנת מדי כדי לבצע אותה אוטומטית. כל אלה הן החלטות מערכתיות שנשארות באחריות מי שבונה את המערכת.

ההבחנה הזו חשובה כי היא מגדירה את חלוקת האחריות. MCP נותן את השפה המשותפת. המערכת שמסביב נותנת את השיפוט, ההגנות, ומדיניות הפעולה.

מתכנת שמצפה ש-MCP יטפל גם בהרשאות גם בלוגים וגם בבטיחות יגלה שהמערכת שלו פתוחה מדי. מתכנת שמבין את הגבולות יתכנן את השכבות הנוספות שהמערכת צריכה מההתחלה.



## שלושת סוגי היכולות: Resources, Tools, Prompts

MCP מגדיר שלושה סוגי יכולות ששרת יכול לחשוף. כל אחת מהן מייצגת כוונה שונה לחלוטין, ובלבול ביניהן הוא אחד הגורמים הנפוצים לבעיות תכנון.

**Resources**

Resource הוא מידע שהמודל יכול לקרוא. קובץ, מסמך, תוצאת שאילתה, סטטוס של תהליך. Resource הוא פסיבי במהותו: הוא חושף נתונים, לא מבצע פעולות. קריאה ל-Resource לא אמורה לשנות שום דבר במערכת. היא רק מחזירה מידע שהמודל יכול להשתמש בו כהקשר.

**Tools**

Tool הוא פעולה שהמודל יכול להפעיל. עדכון רשומה, שליחת הודעה, הפעלת תהליך, מחיקת נתון. Tool הוא אקטיבי במהותו: הוא משנה מצב במערכת. זה מה שהופך אותו לרכיב שדורש תכנון זהיר, ולידציה קפדנית, והרשאות מוגדרות. כל Tool הוא בעצם נקודת גישה לעולם האמיתי.

**Prompts**

Prompt במובן של MCP הוא תבנית מובנית לאינטראקציה. לא פרומפט חופשי שהמשתמש כותב, אלא תבנית מוגדרת מראש שמכוונת את המודל לבצע משימה מסוימת בצורה עקבית. לדוגמה, תבנית לניתוח מסמך משפטי, תבנית לסיכום פגישה, או תבנית לביצוע תהליך עבודה מוגדר.

**למה ההבחנה חשובה**

הטעות הנפוצה ביותר היא להפוך כל דבר ל-Tool. מתכנת שרוצה שהמודל יוכל לקרוא נתון מגדיר Tool שקורא נתון, במקום Resource. זה נראה שקול, אבל זה לא. Tool מרמז על פעולה שמשנה מצב. מי שמשתמש במערכת מצפה ש-Tool יעשה משהו. מי שמתחזק את המערכת מתייחס ל-Tool בזהירות רבה יותר. הגדרת קריאת נתון כ-Resource מבהירה את הכוונה ומפשטת את ניהול ההרשאות.

היכולות האלה הן לא רק קטגוריות טכניות. הן שפה שמאפשרת לצוות לדבר על יכולות המערכת בצורה ברורה ומשותפת.



## MCP ב-2026: היכן הוא פועל היום

MCP עבר מהר מאוד מרעיון של חברה אחת לתקן שהתעשייה מאמצת בפועל.

בסביבות פיתוח, כלים כמו Cursor, VS Code ו-Claude Code תומכים ב-MCP ומאפשרים למפתחים לחבר שרתי MCP ישירות לסביבת העבודה שלהם. מפתח יכול לחבר שרת MCP שמאפשר למודל לגשת למסד הנתונים של הפרויקט, לקרוא תיעוד פנימי, או להפעיל כלי בדיקה, הכל מתוך אותה סביבה שהוא עובד בה.

בצד הארגוני, חברות מתחילות לפרסם שרתי MCP רשמיים לשירותים שלהן. כלי פרודוקטיביות, מסדי נתונים, פלטפורמות ענן, ושירותי SaaS שונים חושפים את היכולות שלהם דרך MCP כדי שמודלים יוכלו לעבוד איתם בצורה מסודרת.

בצד הפרוטוקול עצמו, הגרסה הנוכחית של MCP הוסיפה יכולות שלא היו בגרסה הראשונה. Streamable HTTP החליף את SSE כמנגנון התקשורת המועדף לסביבות Production. SSE היה מנגנון פשוט שעבד טוב בפיתוח מקומי אבל התקשה להתמודד עם עומס אמיתי וחיבורים מרובים. Streamable HTTP פותר את הבעיות האלה ומתאים טוב יותר לסביבה שמערכות אמיתיות דורשות.

בנוסף, נוסף סוג יכולת חדש בשם Tasks. Tasks מאפשר למודל להפעיל פעולה שלוקחת זמן, להמשיך הלאה בלי לחכות לתוצאה, ולחזור לקבל אותה מאוחר יותר. זה קריטי לפעולות כגון: עיבוד קבצים גדולים, הרצת תהליכים ארוכים, או כל משימה שלא מחזירה תוצאה מיידית.

המצב היום הוא ש-MCP הוא תקן פעיל ומתפתח. הוא לא ניסיוני ולא עתידי. הוא נמצא בשימוש בסביבות ייצור אמיתיות, והאקוסיסטם סביבו גדל בקצב מהיר.



## תרגול

בסיום הפרק הזה, אתה אמור להיות מסוגל להסביר מהו MCP בצורה ברורה.

נסה לכתוב הסבר קצר, בין חמישה לעשרה משפטים, שעונה על השאלות הבאות:

1. מה הבעיה ש-MCP- בא לפתור, בלי להשתמש במונח N×M?

2. מה ההבדל בין MCP לבין קריאה רגילה ל-API, בלי להיכנס לפרטים טכניים?

3. מי הם השחקנים במערכת מבוססת MCP ומה תפקיד כל אחד, במשפט אחד לכל שחקן?

4. אילו שלושה סוגי יכולות שרת MCP יכול לחשוף, ומה ההבדל המהותי ביניהם?

אם אתה מסוגל לענות על ארבע השאלות האלה בצורה ברורה, בשפה שמתכנת שמעולם לא שמע על MCP יבין, סיימת את הפרק הזה עם הבנה אמיתית. אם מצאת שאלה שקשה לך לענות עליה בלי לחזור לטקסט, זו בדיוק הנקודה שכדאי לחזור אליה לפני שממשיכים.
