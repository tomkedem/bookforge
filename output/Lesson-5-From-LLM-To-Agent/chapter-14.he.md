# תיקיית data, שיחה אמיתית וסכמה שמכריחה מבנה

עד עכשיו דיברנו בעיקר על הקוד: איך שולחים בקשה למודל, איך בונים API, ואיך מחזירים JSON מובנה.

אבל בשלב הזה מגיע אחד החלקים החשובים ביותר בכל המערכת: הנתונים עצמם.

בתיקיית data נמצאים שני קבצים שממחישים בצורה מאוד טובה את המעבר מטקסט אנושי חופשי למידע שמערכת יכולה לעבוד איתו.

הקובץ הראשון:

call1.txt

מכיל תמלול של שיחה אמיתית בין לקוח לבין נציג ביטוח.

הקובץ השני:

call_summary_schema.json

מכיל את ה-schema שמכריח את המודל להחזיר תשובה במבנה מדויק.

וזו בדיוק הנקודה המרכזית של הפרק:

המודל מקבל כאוס אנושי, אבל המערכת דורשת סדר.

**השיחה עצמה היא טקסט חופשי לחלוטין. אנשים מדברים בצורה לא מסודרת:**

- קופצים בין נושאים

- משנים ניסוחים

- עוצרים באמצע משפט

- מתקנים את עצמם

- מביעים רגש

- שואלים שאלות לא ישירות

- מוסיפים מידע לא רלוונטי

מבחינת מחשב רגיל, זו בעיה קשה מאוד.

**קוד קלאסי אוהב מבנה ברור:**

- עמודות

- שדות

- טיפוסים

- ערכים צפויים

אבל שיחה אנושית לא נראית כך.

לכן במשך שנים ארגונים החזיקו כמויות עצומות של מידע לא מובנה:

- שיחות שירות

- מיילים

- מסמכים

- הערות CRM

- תמלולים

- צ'אטים

המידע היה קיים, אבל קשה מאוד להפוך אותו למשהו שמערכת באמת יכולה להבין.

ופה LLMs משנים את התמונה.

המודל יודע לקרוא טקסט אנושי חופשי, להבין הקשר, לזהות ישויות, להבין רגש, ולהסיק מה באמת קרה בשיחה.

אבל חשוב להבין: 
המודל לבדו עדיין לא מספיק.

אם פשוט נכתוב:

"Summarize this conversation"

נקבל תשובה שנשמעת יפה לבני אדם, אבל לא בהכרח מתאימה למערכת.

לדוגמה:

```bash
The customer called regarding a damaged vehicle claim.
The representative explained the process and scheduled a follow-up.
The customer sounded frustrated but cooperative.
```

זו תשובה נחמדה לקריאה אנושית.

אבל איך מערכת אחרת אמורה לעבוד עם זה?

- איפה מזהה הלקוח?

- איפה סטטוס הטיפול?

- איפה משימות ההמשך?

- איפה סוג האירוע?

- איפה הסנטימנט?

- איפה רשימת המשתתפים?

ופה נכנס הקובץ:

**call_summary_schema.json**

ה-schema לא מבקש מהמודל "לסכם". הוא מגדיר חוזה.

המערכת אומרת למודל:

אתה חייב להחזיר JSON במבנה מדויק.

לדוגמה:

```python
{
  "participants": [],
  "topics": [],
  "outcome": "",
  "follow_up_tasks": [],
  "sentiment": "",
  "call_id": ""
}
```

כלומר:

- לא טקסט חופשי

- לא פסקאות

- לא ניסוחים יצירתיים

- לא מבנה משתנה

אלא אובייקט מוגדר היטב.

המשמעות כאן עמוקה מאוד.

אנחנו לא משתמשים ב-LLM בתור "צ'אט נחמד".

אנחנו משתמשים בו כמנוע המרה:

מטקסט אנושי כאוטי למבנה נתונים יציב.

וזה אחד השימושים העסקיים החשובים ביותר של מודלים כיום.

ה-schema עצמו יכול להיות הרבה יותר מפורט.

לדוגמה:

```python
{
  "type": "object",
  "properties": {
    "call_id": {
      "type": "string"
    },
    "participants": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "name": { "type": "string" },
          "role": { "type": "string" }
        },
        "required": ["name", "role"]
      }
    },
    "topics": {
      "type": "array",
      "items": {
        "type": "string"
      }
    },
    "outcome": {
      "type": "string"
    },
    "follow_up_tasks": {
      "type": "array",
      "items": {
        "type": "string"
      }
    },
    "sentiment": {
      "type": "string"
    }
  },
  "required": [
    "call_id",
    "participants",
    "topics",
    "outcome",
    "follow_up_tasks",
    "sentiment"
  ]
}
```

שימו לב מה קורה כאן.

ה-schema לא רק אומר אילו שדות קיימים.

הוא גם מגדיר:

- אילו שדות חובה

- איזה טיפוס יש לכל שדה

- אילו מבנים פנימיים מותרים

- האם זה מערך או אובייקט

- האם הערך הוא string או boolean

- אילו אובייקטים יכולים להופיע בתוך מערכים

כלומר, אנחנו מכריחים את המודל להתיישר לפי מבנה תוכנתי אמיתי.

אחת ההגדרות החשובות ביותר ב-schema היא:

```python
"additionalProperties": false
```

זו הגדרה קריטית.

היא אומרת:

אסור להחזיר שדות שלא הוגדרו מראש.

למה זה חשוב?

כי LLMs אוהבים "לעזור".

אם לא נגדיר גבולות ברורים, המודל עלול להחזיר:

```python
{
  "call_id": "123",
  "participants": [],
  "topics": [],
  "summary": "Customer was upset",
  "confidence_score": 0.92,
  "agent_notes": "Needs escalation"
}
```

אבל:

- summary לא הוגדר

- confidence_score לא הוגדר

- agent_notes לא הוגדר

מבחינת בן אדם זה אולי נראה מועיל.

מבחינת מערכת תוכנה זו בעיה.

מערכות downstream בונות על מבנה קבוע:

- מסדי נתונים

- pipelines

- dashboards

- analytics

- integrations

- ETL jobs

- BI systems

ברגע שהמבנה מתחיל להשתנות בצורה לא צפויה, מתחילות תקלות.

לכן:

```python
"additionalProperties": false
```

הוא למעשה דרך להגיד למודל:

אל תהיה יצירתי. 
תהיה מדויק.

וזה שינוי תפיסתי חשוב מאוד בעולם ה-AI.

בצ'אט רגיל אנחנו רוצים שהמודל יהיה גמיש ויצירתי.

במערכות production אנחנו בדרך כלל רוצים בדיוק ההפך:

- יציבות

- עקביות

- מבנה קבוע

- חוזה ברור

המודל עדיין משתמש ביכולת ההבנה שלו כדי לנתח את השיחה, אבל הפלט חייב להיכנס לתוך מסגרת קשיחה.

וזה בדיוק מה שהופך LLM ממערכת "שמדברת יפה" לרכיב אמיתי בארכיטקטורת תוכנה.

הנקודה החשובה ביותר בפרק הזה היא שהמודל לא רק "קורא טקסט".

הוא מבצע טרנספורמציה סמנטית.

הוא לוקח:

```python
"I called last week because my car was damaged after the accident and nobody got back to me."
```

ומבין:

- מדובר בתלונת לקוח

- יש עיכוב בטיפול

- יש תסכול

- הנושא הוא claim ביטוחי

- נדרשת פעולה המשך

- ייתכן שיש SLA שהופר

המידע הזה לא הופיע בצורה מובנית בטקסט.

המודל הסיק אותו מתוך משמעות.

וזו אחת הסיבות ש-LLMs כל כך משמעותיים למערכות עסקיות.

הם מאפשרים בפעם הראשונה להפוך כמויות עצומות של טקסט אנושי לא מובנה לנתונים שמערכות באמת יכולות לעבד.

המשמעות העסקית עצומה:

- חיפוש חכם

- אנליטיקה

- CRM enrichment

- ניטור שיחות

- זיהוי סיכונים

- QA אוטומטי

- compliance

- סיכומי פגישות

- ניתוח רגשות

- extraction למסמכים

- triage אוטומטי

- workflows מבוססי AI

והכול מתחיל מאותו רעיון פשוט:

טקסט חופשי נכנס. Schema קשיח יוצא.
