# Tool use, הרגע שבו המודל מקבל יכולת לפעול

עד עכשיו השיעור עסק בעיקר בקריאות למודל, ניהול messages, עבודה עם async, והרצת כמה בקשות במקביל. אבל בשלב הזה המרצה עבר לאחד הנושאים החשובים ביותר בעולם ה-Agents: Tool use.

זהו הרגע שבו מתחילים להבין את הגבול האמיתי של LLM, וגם איך אפשר להרחיב אותו.

המרצה הסביר שמודל שפה בפני עצמו יודע לעשות דבר אחד מרכזי: לקבל טקסט ולהחזיר טקסט. הוא יודע לנתח שפה, להסביר, לסכם, להפיק קוד, לענות על שאלות, ולהחזיר פלט שנראה חכם מאוד. אבל הוא אינו מחובר מעצמו לעולם האמיתי.

**כלומר, גם אם מודל כותב:**

“בדקתי את מזג האוויר” - זה לא אומר שהוא באמת ביצע בדיקה אמיתית.

**וגם אם הוא כותב:**

“קבעתי לך פגישה ביומן” - זה לא אומר שבאמת נוצר אירוע ביומן.

המודל עצמו לא ניגש לבד למסד נתונים.

הוא לא מתקשר לבד ל-API חיצוני.

הוא לא מפעיל שירותים.

והוא לא מבצע פעולות בעולם האמיתי.

כדי שזה יקרה, צריך לתת לו tool.

וזו נקודת המעבר החשובה.

## מהו בעצם Tool?

Tool הוא פעולה חיצונית שהאפליקציה מאפשרת למודל להשתמש בה.

המודל לא מפעיל את ה-tool ישירות. במקום זאת, הוא מקבל תיאור של הכלים הזמינים, ובמידת הצורך הוא יכול “לבקש” להשתמש באחד מהם.

האפליקציה היא זו שמבצעת בפועל את הפעולה.

לאחר מכן, התוצאה חוזרת למודל כחלק מה-Context, והמודל ממשיך את השיחה או מייצר תשובה סופית למשתמש.

**כלומר, Tool use יוצר חלוקה ברורה:**

המודל מחליט איזה מידע או פעולה נחוצים.

האפליקציה מבצעת את הפעולה בפועל.

המודל מקבל את התוצאה וממשיך לחשוב עליה.

## דוגמאות ל-tools

המרצה נתן כמה דוגמאות פשוטות כדי להמחיש את הרעיון.

**בדיקת מזג אוויר**

נניח שהמשתמש שואל: “מה מזג האוויר בתל אביב?”

המודל עצמו לא באמת יודע את מזג האוויר הנוכחי. הוא לא מחובר ישירות לשירות מטאורולוגי.

אבל אם האפליקציה מספקת tool בשם למשל:

get_weather(city)

המודל יכול לבקש להשתמש בו.

האפליקציה תפעיל API אמיתי של מזג אוויר, תחזיר את הנתונים, ואז המודל יוכל לנסח תשובה טבעית למשתמש.

**קריאה למסד נתונים**

נניח שמשתמש שואל: “כמה הזמנות פתוחות יש לי?”

המודל לא יכול לפתוח לבד חיבור למסד נתונים ארגוני.

אבל אפשר לספק לו tool כמו:

get_open_orders(customer_id)

האפליקציה תבצע את השאילתה בפועל, תחזיר את הנתונים, והמודל יוכל להסביר אותם למשתמש בשפה טבעית.

**קביעת אירוע ביומן**

גם כאן, המודל לא באמת פותח Google Calendar לבד.

אבל אם קיים tool כמו:

create_calendar_event(...)

המודל יכול לבקש יצירת פגישה.

האפליקציה מבצעת את הפעולה מול מערכת היומן, ואז מחזירה למודל אישור שהאירוע נוצר.

**קריאה ל-API פנימי**

במערכות ארגוניות זה אחד השימושים החשובים ביותר.

המודל יכול להשתמש ב-tools כדי:

- לבדוק סטטוס משלוח

- לפתוח קריאת שירות

- לשלוף נתוני לקוח

- לעדכן מידע במערכת פנימית

- להפעיל workflow

- או לבצע פעולות מול microservices אחרים

כלומר, ה-LLM הופך לשכבת הבנה ושיחה מעל מערכות קיימות.

**שליפת מידע ממערכת ארגונית**

זו אחת הנקודות שבהן Tool use מתחיל להתחבר לעולם האמיתי של AI בתוך ארגונים.

במקום להכניס למודל את כל המידע מראש, אפשר לתת לו כלים שמאפשרים לשלוף מידע בזמן אמת לפי הצורך.

למשל:

```python
search_customer_documents(...)
```

או:

```python
get_task_status(...)
```

המודל יכול להחליט מתי כדאי לקרוא לכלי, לקבל את הנתונים, ואז להשתמש בהם כדי לענות בצורה מדויקת יותר.



## למה Tool use כל כך חשוב?

כאן מתחיל המעבר האמיתי מ “מודל שמחזיר טקסט” אל “מערכת שמבצעת עבודה”.

בלי tools, המודל מוגבל למה שכבר נמצא בתוך ה-Context או בתוך הידע שעליו אומן.

עם tools, הוא יכול להתחיל לעבוד מול העולם החיצוני.

זו נקודה קריטית להבנת Agents.

Agent אינו רק “LLM חכם”.

Agent הוא LLM שיודע להשתמש בכלים.

הוא יודע:

- לבקש מידע

- להפעיל פעולה

- לקבל תוצאה

- ולהמשיך לעבוד על סמך התוצאה הזו

במילים אחרות, tools הם הידיים של ה-Agent.

## איפה נכנס MCP?

בשלב הזה המרצה הזכיר בזהירות גם את MCP, אבל בלי להיכנס לעומק עדיין. הוא ציין שזה נושא חשוב מאוד להמשך, משום שהוא קשור לדרך מסודרת שבה מודלים יכולים לעבוד מול כלים, משאבים ושירותים חיצוניים.

כרגע המטרה אינה ללמוד MCP עצמו, אלא להבין קודם את הרעיון הבסיסי:

המודל לא פועל לבד.

הוא זקוק לשכבת tools שהאפליקציה מספקת לו.

רק אחרי שמבינים את המנגנון הזה בצורה פשוטה, אפשר להתחיל לדבר על פרוטוקולים ומערכות שמנהלות אותו בצורה רחבה יותר.

## החיבור למה שכבר למדנו

Tool use מתחבר כמעט לכל מה שכבר הופיע בשיעור:

**messages** - כי גם תוצאות ה-tool חוזרות כחלק מה-Context.

**Structured Output** - כי המודל צריך להחזיר בקשה מסודרת להפעלת tool.

**async** - כי הרבה tools עובדים דרך APIs חיצוניים ולכן מתאימים מאוד לעבודה אסינכרונית.

**Debugging** ו-**correlationId** - כי עכשיו כבר לא עוקבים רק אחרי קריאה למודל, אלא גם אחרי tools שהופעלו במהלך הדרך.

כלומר, Tool use אינו נושא נפרד. הוא הנקודה שבה כל החלקים הקודמים מתחילים להתחבר למערכת אחת.

והמסר המרכזי של הפרק הוא פשוט מאוד:

LLM יודע לחשוב על פעולה.

Tool מאפשר למערכת לבצע אותה בפועל.
