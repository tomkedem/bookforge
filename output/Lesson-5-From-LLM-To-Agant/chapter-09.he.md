# המעבר מארגז תרגול לארכיטקטורה של Agents

אחרי שהמרצה הסביר את הרעיון של Tool use והציג את ה-Agentic loop בקוד, השיעור עבר שלב.

עד עכשיו ראינו את המנגנון ברמת קובץ Python: איך מגדירים tool, איך המודל מבקש להשתמש בו, איך האפליקציה מפעילה אותו, ואיך מחזירים את התוצאה למודל.

אבל במערכת אמיתית לא בונים את כל זה כקובץ בודד שמריצים מקומית. בשלב הזה המרצה עבר למצגת והציג את הרעיון של **Emerging Agentic Architecture.**

כלומר, איך נראה המבנה הרחב יותר של מערכת ארגונית שמוסיפה יכולות AI ו-Agents מעל אפליקציות קיימות.

הרעיון המרכזי הוא שלא כל אפליקציה בארגון אמורה לקרוא ישירות ל-LLM בעצמה. אם כל מערכת תתחבר לבד למודל, תנהל לבד prompts, תנהל לבד context, תפעיל לבד tools ותטפל לבד בשגיאות, מהר מאוד נקבל ארכיטקטורה מבולגנת, יקרה וקשה לניטור.

במקום זה, בונים שכבה ייעודית שמרכזת את העבודה מול המודל.

בשיעור היא הוצגה כשכבת App Agent, Agent Hub, או בשם דומה.

זו שכבה שיושבת בין האפליקציות העסקיות לבין מודלי ה-LLM והשירותים החיצוניים.

בחלק העליון נמצאות האפליקציות הקיימות: CRM, מערכת Support, מערכת Analytics, כלי פיתוח או כל מערכת עסקית אחרת. אלו המערכות שבהן המשתמשים עובדים בפועל. הן לא בהכרח יודעות איך לדבר עם מודל, איך לנהל context או איך להפעיל tools. הן פשוט צריכות יכולת AI שתעזור להן לבצע פעולה מסוימת.

מתחתיהן נמצאת שכבת ה-Agent Hub. זו שכבה שמרכזת Agents לפי תחומים:

למשל Sales Agent, Support Agent, Data Agent או DevOps Agent. כל Agent כזה יודע לעבוד בהקשר עסקי מסוים, עם כלים מסוימים, ועם סוגי משימות מסוימים.

מתחת לשכבת ה-Agents מופיעה שכבת תשתית משותפת. זו שכבה חשובה מאוד, כי היא כוללת את הדברים שכל Agent צריך, בלי שכל צוות ימציא אותם מחדש: orchestration, memory, tool registry, guardrails, observability ועוד. במילים אחרות, זו שכבת התפעול והבקרה של עולם ה-Agents.

לאחר מכן מגיעה שכבת LLM Gateway. התפקיד שלה הוא לרכז את העבודה מול ספקי המודלים. במקום שכל Agent יקרא ישירות ל-OpenAI, Anthropic או Gemini,

יש שער מרכזי שיכול לבצע routing בין מודלים, caching, rate limiting, cost tracking וניהול מדיניות. כך הארגון יכול לשלוט טוב יותר בעלויות, בביצועים, באבטחת מידע ובגמישות מול ספקים שונים.

בתחתית נמצאים שני סוגי משאבים: ספקי מודלים ושירותים חיצוניים.

ספקי המודלים יכולים להיות למשל OpenAI, Anthropic או Gemini. השירותים החיצוניים יכולים להיות CRM, מסדי נתונים, APIs, vector stores או כל מערכת ארגונית שממנה צריך לשלוף מידע או שבה צריך לבצע פעולה.

המסר המרכזי מהאיור הוא ש-Agentic Architecture אינה רק “מודל חכם יותר”. זו ארכיטקטורה שלמה סביב המודל.

המודל הוא רכיב חשוב, אבל הוא לא כל המערכת.

מערכת כזו צריכה לדעת:

- לנהל context

- לתזמר כמה צעדים

- להפעיל tools

- לשלוף מידע ממערכות חיצוניות

- להחזיר תוצאות למודל

- לבצע routing בין מודלים

- לעקוב אחרי עלויות

- להוסיף guardrails

- לנטר תקלות

- ולאפשר לאפליקציות קיימות לצרוך יכולות AI דרך שכבה מסודרת

זו בדיוק נקודת המעבר מארגז תרגול לארכיטקטורה אמיתית. בקובץ 7_anthropic_api_tool_use.py ראינו loop קטן ופשוט. באיור זה רואים איך אותו רעיון מתרחב למערכת שלמה: לא tool אחד, אלא registry של tools.

לא קריאה אחת למודל, אלא gateway. לא session מקומי, אלא memory ותשתית משותפת. לא אפליקציה אחת, אלא כמה מערכות עסקיות שצורכות יכולות AI מאותה שכבה.
