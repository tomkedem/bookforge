# Agentic loop לפי 7_anthropic_api_tool_use.py

בפרק הקודם הכרנו את הרעיון של Tool use: המודל אינו מבצע פעולות בעצמו, אלא מבקש מהאפליקציה להפעיל עבורו כלים. עכשיו השיעור עובר לשלב המעשי באמת: איך זה נראה בקוד.

הקובץ 7_anthropic_api_tool_use.py הוא אחד הקבצים החשובים בשיעור 5, משום שכאן מופיע לראשונה מה שאפשר לקרוא לו Agentic loop - לולאה שבה המודל חושב, מבקש להשתמש בכלי, מקבל תוצאה, וממשיך לעבוד עד להפקת תשובה סופית למשתמש.

זהו כבר לא מודל שמקבל prompt ומחזיר תשובה אחת.

זהו תהליך רב שלבי.

הזרימה הכללית נראית כך:

וזה בדיוק הרגע שבו מתחילים להבין מהו Agent בפועל: לא קסם, אלא לולאה שמחברת בין מודל, כלים, ותוצאות.

## הגדרת ה-tool

הקובץ מתחיל בהגדרת tool בשם:

```python
create_calendar_event
```

זהו כלי שמטרתו לדמות יצירת אירוע ביומן.

המודל עצמו כמובן לא פותח באמת Google Calendar. הכלי כאן הוא פונקציה בצד האפליקציה, שהמודל יכול לבקש להשתמש בה.

ההגדרה כוללת כמה חלקים חשובים:

```python
{
    "name": "create_calendar_event",
    "description": "Create a calendar event",
    "input_schema": {
        ...
    }
}
```

**name** הוא שם ה-tool.

זהו השם שבו המודל ישתמש כאשר ירצה להפעיל את הכלי.

**description** מסביר למודל מה הכלי עושה. זה חשוב מאוד, כי המודל מחליט באיזה tool להשתמש לפי התיאור שניתן לו.

אם התיאור לא ברור, המודל עלול לבחור tool לא מתאים או להשתמש בו בצורה שגויה.

מהו **input_schema**

החלק החשוב ביותר בהגדרת ה tool הוא:

input_schema

זהו המבנה שמסביר למודל איזה קלט הכלי מצפה לקבל.

למשל:

```python
"input_schema": {
    "type": "object",
    "properties": {
        "title": {"type": "string"},
        "date": {"type": "string"},
        "time": {"type": "string"}
    },
    "required": ["title", "date", "time"]
}
```

הסכמה הזו אומרת למודל:

אם אתה רוצה להשתמש ב-create_calendar_event, אתה צריך לספק:

- כותרת

- תאריך

- שעה

כלומר, ה-tool לא מקבל טקסט חופשי.

הוא מקבל מבנה מסודר.

זו נקודה חשובה מאוד. ברגע שמתחילים לעבוד עם tools, המודל צריך להחזיר מידע בצורה שהאפליקציה יכולה באמת להשתמש בה.

כאן רואים שוב את החשיבות של Structured Output, שכבר הופיעה בתרגיל סיווג המסמכים.

## שליחת ה-tools למודל

כאשר האפליקציה קוראת ל-Claude, היא לא שולחת רק messages.

היא שולחת גם רשימת tools.

כלומר, המודל מקבל לא רק את השיחה, אלא גם מידע על הכלים שזמינים לו.

זה רגע חשוב להבנה:

המודל לא “מכיר” tools מראש.

ה-tools קיימים רק משום שהאפליקציה סיפקה אותם כחלק מהבקשה.

אפשר לחשוב על זה כאילו האפליקציה אומרת למודל:

```python
Here are the tools you are allowed to use.
```

והמודל מחליט האם כדאי להשתמש בהם.

**tool_choice - בחירה אוטומטית של tool**

בקוד מופיע גם:

```python
tool_choice={"type": "auto"}
```

המשמעות היא שהמודל עצמו מחליט האם צריך tool, ואם כן - איזה tool להפעיל.

זו נקודה מעניינת מאוד.

האפליקציה לא אומרת:

“עכשיו תפעיל את create_calendar_event”

במקום זאת, המודל קורא את בקשת המשתמש, רואה אילו tools זמינים, ומחליט בעצמו האם צריך להשתמש באחד מהם.

למשל, אם המשתמש כותב:

```python
Schedule a meeting tomorrow at 10 AM
```

Claude יכול להבין שזה דורש כלי של יומן.

```python
stop_reason == "tool_use"
```

אחרי שהמודל מחזיר תשובה, הקוד בודק:

```python
response.stop_reason == "tool_use"
```

זהו אחד החלקים החשובים ביותר בכל הלולאה.

stop_reason מסביר למה המודל עצר את יצירת התשובה.

במקרה רגיל, המודל פשוט מחזיר טקסט.

אבל כאשר הוא רוצה להשתמש ב tool, הוא לא מחזיר תשובה סופית למשתמש. במקום זאת, הוא עוצר ומחזיר בקשה להפעלת tool.

כלומר:

```python
I need the application to execute a tool now.
```

זו למעשה נקודת המעבר בין “חשיבה של המודל” לבין “ביצוע פעולה באפליקציה”.



## חילוץ בקשת ה-tool

כאשר stop_reason הוא "tool_use", הקוד מחלץ מתוך התשובה:

- איזה tool המודל ביקש

- אילו פרמטרים הוא שלח

- מהו ה input של ה-tool

<img src="/Lesson-5-From-LLM-To-Agent/assets/image-08.png" alt="image-08.png" width="710" height="538" />



למשל:

```python
tool_name = tool_use.name
tool_input = tool_use.input
```

כאן האפליקציה כבר לא מדברת עם המודל. 
היא עוברת לבצע פעולה אמיתית בצד שלה.

## הרצת הפונקציה בצד האפליקציה

בשלב הזה האפליקציה מפעילה פונקציה רגילה ב Python.

למשל:

```python
create_calendar_event(...)
```

וזו נקודה קריטית להבנה:

המודל עצמו לא יוצר את האירוע.

המודל רק ביקש להשתמש ב tool.

האפליקציה היא זו שמבצעת בפועל את הפעולה.

זה יכול להיות:

- קריאה למסד נתונים

- API חיצוני

- יצירת אירוע ביומן

- שליחת אימייל

- קריאה למערכת ארגונית

- או כל פעולה אחרת שהמפתח מאפשר

## החזרת tool_result למודל

אחרי שהאפליקציה מסיימת להריץ את ה-tool, היא לא מחזירה מיד תשובה למשתמש.

במקום זאת, היא מחזירה את תוצאת ה-tool חזרה ל-Claude.

זה נעשה באמצעות message נוסף מסוג:

```python
tool_result
```

כלומר, המודל מקבל עכשיו מידע חדש:

```bash
Calendar event created successfully
```

או:

```bash
Meeting scheduled for tomorrow at 10 AM
```

**המשך הלולאה**

עכשיו קורה הדבר החשוב באמת.

Claude מקבל את תוצאת ה-tool כחלק מה-Context, וממשיך את התהליך.

הוא יכול עכשיו לייצר תשובה סופית למשתמש, למשל:

```bash
Your meeting has been scheduled for tomorrow at 10 AM.
```

וזו בדיוק הסיבה שקוראים לזה Agentic loop.

**החיבור לעולם ה-Agents**

הפרק הזה הוא למעשה הבסיס לכל עולם ה-Agents.

Agent מודרני אינו רק prompt גדול.

הוא מערכת שיודעת:

- להבין מטרה

- להחליט איזה tool דרוש

- להפעיל פעולה

- לקבל תוצאה

- ולהמשיך לעבוד לפי התוצאה

ובדיוק כאן מתחילים להבין למה tools הם הלב של Agentic AI.

בלי tools, המודל מוגבל למה שהוא כבר יודע.

עם tools, הוא מתחיל לעבוד מול העולם החיצוני.
