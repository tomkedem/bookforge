# עקרונות Prompt Design שעלו בסוף השיעור

לקראת סוף השיעור המרצה עצר לכמה דקות כדי לדבר על Prompt Design.

לא בתור "אמנות קסומה", ולא בתור אוסף טריקים ויראליים, אלא בצורה הרבה יותר פרקטית:

איך כותבים prompts שעובדים טוב במערכות אמיתיות.

הדגש היה חשוב: 
אין נוסחת קסם.

Prompt טוב הוא בדרך כלל:

- ברור

- ממוקד

- מדויק

- מותאם למשימה

ובעיקר: 
קל למודל להבין מה באמת רוצים ממנו.

## להיות תמציתי

אחת הטעויות הנפוצות ביותר היא לחשוב שיותר טקסט תמיד משפר תוצאות.

בפועל, הרבה prompts גרועים פשוט עמוסים מדי.

לדוגמה:

```python
You are the best AI assistant in the world with amazing reasoning capabilities...
```

ולאחר מכן עוד כמה פסקאות ארוכות שלא באמת מוסיפות מידע שימושי.

מודלים לא צריכים מחמאות.

הם צריכים הוראות ברורות.

ברוב המקרים prompt קצר וברור יעבוד טוב יותר מ-prompt מנופח.

לדוגמה:

```python
Extract the customer issue and sentiment from this conversation.
Return valid JSON only.
```

זה ברור. 
ממוקד. 
קל להבין מה המשימה.

## להימנע מעומס context

עוד context לא תמיד עוזר.

לפעמים הוא אפילו פוגע.

כאשר דוחפים למודל:

- יותר מדי דוגמאות

- היסטוריה ארוכה מדי

- מידע לא רלוונטי

- הוראות כפולות

- ניסוחים סותרים

המודל עלול:

- להתבלבל

- להתמקד בפרטים הלא נכונים

- להחמיץ מידע חשוב

- לייצר תשובות פחות עקביות

זו נקודה חשובה במיוחד ב-multi-turn systems.

לא כל ההיסטוריה תמיד רלוונטית.

מערכות AI טובות לומדות:

- מה להכניס ל-context

- ומה לא



## להימנע מחזרתיות

לפעמים מפתחים כותבים שוב ושוב את אותה הוראה:

```python
Return JSON only.
Do not return explanations.
Only JSON.
No extra text.
```

קצת חיזוק יכול לעזור.

אבל חזרתיות מוגזמת לא בהכרח משפרת תוצאות. 
לפעמים היא אפילו מוסיפה רעש.

Prompt טוב בדרך כלל נשמע כמו specification ברור, לא כמו אדם לחוץ שחוזר על עצמו שוב ושוב.

## לשים מידע קריטי בתחילת או סוף הפרומפט

זו נקודה מעניינת שעלתה בשיעור.

ב-prompts ארוכים יחסית, המיקום של המידע משנה.

בפועל, מודלים נוטים לתת יותר משקל:

- למה שמופיע בהתחלה

- ולמה שמופיע ממש בסוף

לכן אם יש הוראה קריטית, עדיף שלא לקבור אותה באמצע paragraph ארוך.

לדוגמה:

```python
IMPORTANT:
Return valid JSON matching the schema exactly.
```

עדיף למקם במקום בולט וברור.

## להעדיף פורמט ברור במשימות מורכבות

כאשר המשימה מתחילה להיות מורכבת, טקסט חופשי נהיה קשה לקריאה גם עבור בני אדם וגם עבור מודלים.

לכן הרבה מערכות משתמשות בפורמטים ברורים יותר כמו:

- Markdown

- XML

- JSON

- YAML

לדוגמה:

```python
<task>
    Extract customer complaint details
</task>

<rules>
    Return valid JSON only
</rules>
```

או:

```python
## Task
Extract shipping issues from the conversation.

## Output Format
Return valid JSON.

## Rules
- No explanations
- No extra fields
```

המטרה היא לא "לקשט" את הפרומפט.

המטרה היא להפריד בצורה ברורה בין:

- משימה

- חוקים

- דוגמאות

- פורמט פלט

- context

וזה נהיה חשוב יותר ככל שהמערכת גדלה.

## לפרק משימות מורכבות

עוד טעות נפוצה: 
לבקש מהמודל לעשות את הכול בבת אחת.

לדוגמה:

```python
Analyze the conversation, detect fraud, summarize the call,
extract entities, calculate risk, classify the customer,
and generate follow-up tasks.
```

לפעמים זה יעבוד.

אבל בהרבה מקרים עדיף לפרק את הבעיה.

לדוגמה:

1. חילוץ ישויות

2. סיווג שיחה

3. ניתוח סנטימנט

4. יצירת משימות המשך

כל שלב קטן וברור יותר.

וזה בדרך כלל משפר:

- יציבות

- דיוק

- debugging

- observability

- יכולת בדיקה

זו גישה מאוד נפוצה במערכות agentic מודרניות.

## לבדוק מודלים שונים

לא כל מודל טוב באותן משימות.

יש מודלים שטובים יותר ב:

- reasoning

- structured output

- coding

- extraction

- summarization

- multilingual tasks

ולכן חשוב לבדוק.

אותו prompt בדיוק יכול להתנהג שונה מאוד בין:

- Claude

- GPT

- Gemini

- מודלים פתוחים

לפעמים שינוי מודל משפר תוצאה יותר מכל שינוי prompt.

## להמשיך לבדוק

עוד נקודה מעניינת שעלתה בשיעור: 
Prompt Engineering הוא תחום מאוד דינמי.

מה שלא עבד לפני חצי שנה, לפעמים עובד מצוין היום.

למה?

כי המודלים עצמם משתפרים.

הרבה prompts ישנים נכתבו כדי "לעזור" למודלים חלשים יותר:

- chain of thought ידני

- חזרתיות מוגזמת

- פורמטים מסובכים

- טריקים שונים

אבל מודלים חדשים לעיתים כבר לא צריכים חלק גדול מזה.

לכן חשוב לא להינעל על assumptions ישנים.

Prompt טוב ב-2024 לא בהכרח יהיה prompt טוב ב-2026.

## הנקודה המרכזית

בסופו של דבר Prompt Design הוא לא קסם. זו פשוט דרך לתקשר בצורה ברורה עם מערכת הסתברותית.

Prompt טוב:

- מצמצם ambiguity

- מגדיר ציפיות

- מגדיר גבולות

- עוזר למודל להבין מה חשוב

- ומקל על המערכת לקבל פלט יציב יותר

אבל גם ה-prompt הכי טוב בעולם לא מחליף:

- validation

- retries

- schema enforcement

- business rules

- defensive programming

וזה היה המסר החשוב בסיום השיעור: Prompt הוא רק שכבה אחת בתוך מערכת AI שלמה.
