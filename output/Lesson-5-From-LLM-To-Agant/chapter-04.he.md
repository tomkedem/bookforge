# Async API מול Anthropic

אחרי החזרה על שיעורי הבית וההסבר למה לא קופצים מיד ל-frameworks מורכבים, המרצה עבר לנושא חדש: עבודה אסינכרונית מול Anthropic API.

הקובץ שנדון בחלק הזה הוא 6_anthropic_api_async.py. זהו קובץ קצר מאוד, אבל הוא פותח נושא חשוב: איך קוראים למודל בצורה אסינכרונית, כלומר בצורה שמתאימה לעבודה מול שירות חיצוני שדורש זמן המתנה לתשובה.

נציג קודם את הקובץ בשלמותו, ואז נפרק אותו בהדרגה.

```python
import os
import asyncio
from anthropic import AsyncAnthropic, DefaultAioHttpClient


async def main() -> None:
    async with AsyncAnthropic(
        api_key=os.environ.get("ANTHROPIC_API_KEY"),
        http_client=DefaultAioHttpClient(),
    ) as client:
        message = await client.messages.create(
            max_tokens=1024,
            messages=[
                {
                    "role": "user",
                    "content": "Hello, Claude",
                }
            ],
            model="claude-haiku-4-5",
        )
        print(message.content)


asyncio.run(main())
```

הקובץ הזה עושה פעולה אחת פשוטה: הוא שולח הודעה אחת ל Claude, מקבל תשובה, ומדפיס אותה למסך. אבל למרות שהפעולה פשוטה, צורת הכתיבה כאן חשובה מאוד, כי היא מציגה את הדרך לעבוד מול Anthropic בצורה אסינכרונית.



**ייבוא הספריות**

```python
import os
import asyncio
from anthropic import AsyncAnthropic, DefaultAioHttpClient
```

**השורה הראשונה:**

```python
import os
```

מייבאת את המודול os, שמאפשר לעבוד מול מערכת ההפעלה. כאן משתמשים בו כדי לקרוא את מפתח ה-API מתוך משתנה סביבה:

```python
os.environ.get("ANTHROPIC_API_KEY")
```

זו צורת עבודה נכונה יותר מאשר לכתוב את המפתח ישירות בקוד. מפתח API הוא מידע רגיש, ולכן הוא צריך להגיע מקובץ סביבה, משתנה סביבה או מנגנון סודות אחר, ולא להיות hardcoded בתוך הקובץ.

**השורה השנייה:**

```python
import asyncio
```

מייבאת את ספריית asyncio, שהיא הספרייה המרכזית של Python לעבודה אסינכרונית.

היא מספקת את מנגנון ה-event loop שמריץ פונקציות async, מנהל המתנות, ומאפשר לקוד להתקדם בצורה יעילה כאשר יש פעולות שממתינות לרשת או לשירות חיצוני.

**השורה השלישית:**

```python
from anthropic import AsyncAnthropic, DefaultAioHttpClient
```

מייבאת שני רכיבים מתוך הספרייה של Anthropic.

AsyncAnthropic הוא ה-client האסינכרוני של Anthropic. כלומר, זה האובייקט שדרכו נשלחות קריאות ל-API, אבל בגרסה שתומכת ב-await.

DefaultAioHttpClient הוא Client HTTP אסינכרוני שמבצע בפועל את בקשות הרשת מאחורי הקלעים. במילים פשוטות, AsyncAnthropic הוא ה-client ברמת Anthropic, ו-DefaultAioHttpClient הוא רכיב התקשורת שמאפשר לו לעבוד בצורה אסינכרונית.

## הגדרת הפונקציה הראשית

```python
async def main() -> None:
```

כאן מוגדרת פונקציה בשם main, אבל זו אינה פונקציה רגילה. המילה async אומרת שזו פונקציה אסינכרונית.

פונקציה אסינכרונית לא רצה מיד כמו פונקציה רגילה. כאשר מפעילים אותה, מתקבלת פעולה אסינכרונית שמיועדת לרוץ בתוך event loop. לכן בהמשך הקובץ לא מפעילים אותה כקריאה רגילה, אלא מריצים אותה באמצעות:

```python
asyncio.run(main())
```

הסימון:

```python
-> None
```

הוא type hint. הוא אומר שהפונקציה לא אמורה להחזיר ערך. היא מבצעת פעולה: שולחת בקשה, מקבלת תשובה, מדפיסה למסך, ומסתיימת.

## יצירת client אסינכרוני

async with AsyncAnthropic( 
 api_key=os.environ.get("ANTHROPIC_API_KEY"), 
 http_client=DefaultAioHttpClient(), 
) as client:

זו אחת השורות החשובות בקובץ.

כאן יוצרים client של Anthropic, כלומר אובייקט שמרכז את התקשורת מול Anthropic API.

חשוב להבין למה בכלל צריך client.

ה-client אינו רק משתנה טכני. הוא השכבה שדרכה הקוד מדבר עם השירות החיצוני. הוא מחזיק את מפתח ה-API, את תצורת התקשורת, את Client ה-HTTP, ואת המתודות שמאפשרות לשלוח בקשות למודל, למשל:

```python
client.messages.create(...)
```

במקום שכל קריאה תבנה לבד בקשת HTTP מלאה,

עם headers, authentication, body, endpoint ו-parsing, הספרייה נותנת לנו client מסודר שעוטף את כל זה.

בתוך יצירת ה client מופיע:

```python
api_key=os.environ.get("ANTHROPIC_API_KEY")
```

כאן הקוד קורא את מפתח ה-API ממשתני הסביבה. אם המפתח מוגדר במערכת, הוא יועבר ל-client, וה-client יוכל לבצע קריאות מזוהות ל-Anthropic.

לאחר מכן מופיע:

```python
http_client=DefaultAioHttpClient()
```

כאן מגדירים ל-client להשתמש ב-HTTP client אסינכרוני. זהו החלק שמאפשר לבקשות הרשת להתנהל בצורה שמתאימה ל-async.



## למה async with?

הקוד לא כותב רק:

```python
client = AsyncAnthropic(...)
```

אלא משתמש ב:

```python
async with AsyncAnthropic(...) as client:
```

הסיבה היא שה-client הוא משאב שיכול לפתוח חיבורי רשת או להחזיק session פנימי. כאשר עובדים עם משאבים כאלה, רצוי לפתוח ולסגור אותם בצורה מסודרת.

async with הוא context manager אסינכרוני. הוא אומר:

פתח את המשאב.

תן לי להשתמש בו בתוך הבלוק.

כשמסיימים, סגור אותו בצורה נקייה.

מכיוון שמדובר במשאב אסינכרוני, למשל כזה שקשור לחיבורי HTTP, משתמשים ב-async with ולא ב-with רגיל.

זו צורת כתיבה חשובה במערכות אמיתיות, כי היא מפחיתה סיכון לדליפות משאבים, חיבורים פתוחים, או התנהגות לא מסודרת של client לאחר סיום השימוש.

## הקריאה למודל

בתוך הבלוק של ה-client מופיעה הקריאה עצמה:

```python
message = await client.messages.create(
    max_tokens=1024,
    messages=[
        {
            "role": "user",
            "content": "Hello, Claude",
        }
    ],
    model="claude-haiku-4-5",
)
```

כאן הקוד שולח בקשה ל-Messages API של Anthropic.

המתודה:

```python
client.messages.create(...)
```

יוצרת הודעה חדשה מול המודל. במילים אחרות, זו הקריאה שבה שולחים למודל את הקלט ומבקשים ממנו להחזיר תשובה.

הפרמטר:

```python
max_tokens=1024
```

קובע את האורך המקסימלי של התשובה שהמודל יכול להחזיר. זו דרך לשלוט בגודל הפלט, בעלויות ובגבולות התשובה.

הפרמטר:

```python
messages=[
    {
        "role": "user",
        "content": "Hello, Claude",
    }
]
```

הוא רשימת ההודעות שנשלחת למודל. במקרה הזה זו רשימה קצרה מאוד, עם הודעה אחת בלבד מהמשתמש.

יש כאן הודעה בתפקיד:

```python
"role": "user"
```

והתוכן שלה הוא:

```python
"content": "Hello, Claude"
```

כלומר, זו קריאה פשוטה מאוד: המשתמש אומר למודל “Hello, Claude”.

הפרמטר:

```python
model="claude-haiku-4-5"
```

קובע באיזה מודל משתמשים. במקרה הזה הקוד משתמש במודל claude-haiku-4-5.

## מה עושה await?

המילה החשובה ביותר בשורת הקריאה היא:

```bash
await
```

השורה המלאה היא:

```python
message = await client.messages.create(...)
```

await אומר לפייתון: הקריאה הזו היא פעולה אסינכרונית, ולכן צריך להמתין לתוצאה שלה.

אבל חשוב לדייק. await לא אומר שכל המערכת נעצרת ולא יכולה לעשות דבר. הוא אומר שהפעולה האסינכרונית הנוכחית ממתינה לתוצאה. בזמן ההמתנה הזה, ה-event loop יכול להמשיך לקדם משימות async אחרות שקיימות במערכת.

בדוגמה הזו יש רק קריאה אחת, ולכן לא רואים עדיין את היתרון הגדול של async. אבל מבחינה רעיונית, זה מכין אותנו לקובץ הבא, שבו יופעלו כמה קריאות למודל במקביל.

הסיבה שקריאה למודל מתאימה ל-async היא שזו קריאה לשירות חיצוני. רוב הזמן הקוד אינו מבצע חישוב מקומי, אלא ממתין לתשובה מהרשת. בזמן המתנה כזה, אין סיבה עקרונית לחסום את כל התוכנית.

במערכת אמיתית, בזמן שקריאה אחת מחכה לתשובת LLM, המערכת יכולה לטפל בבקשה אחרת, לקרוא ל-API אחר, לגשת למסד נתונים, או להמשיך פעולה אחרת שאינה תלויה בתוצאה המיידית של הקריאה הראשונה.

## הדפסת התשובה

אחרי שהתשובה חוזרת מהמודל, הקוד מדפיס אותה:

```python
print(message.content)
```

message הוא האובייקט שחזר מה-API. בתוך האובייקט הזה נמצא התוכן שהמודל החזיר.

בדוגמה הזו מדפיסים את message.content כפי שהוא. זה מתאים להדגמה, כי המטרה היא לראות שהקריאה הצליחה ושהתקבלה תשובה מהמודל.

בקוד Production, בדרך כלל לא היינו מסתפקים בהדפסה. היינו בודקים את מבנה התשובה, מחלצים ממנה את הטקסט, מטפלים בשגיאות, מוסיפים לוגים, ואולי מחזירים את התוצאה ל-API client או לשכבה אחרת במערכת.

## הפעלת התוכנית

בסוף הקובץ מופיעה השורה:

```python
asyncio.run(main())
```

זו השורה שמפעילה את כל התהליך.

כיוון ש main היא פונקציה אסינכרונית, אי אפשר להפעיל אותה כמו פונקציה רגילה. צריך event loop שיריץ אותה.

asyncio.run(main()) עושה כמה דברים:

הוא יוצר event loop.

הוא מריץ את הפונקציה main.

הוא מאפשר ל await לעבוד.

הוא ממתין עד שהפונקציה מסתיימת.

ולבסוף הוא סוגר את ה-event loop בצורה מסודרת.

## למה מתחילים דווקא מדוגמה פשוטה?

הקובץ הזה לא מנסה להראות מערכת מורכבת. להיפך, הוא מציג את הצורה המינימלית ביותר של קריאה אסינכרונית למודל.

וזו בדיוק הסיבה שהוא חשוב.

לפני שמריצים כמה קריאות במקביל, לפני שמשתמשים ב-tools, לפני שעוטפים את המודל ב-FastAPI, צריך להבין את המבנה הבסיסי:

יש פונקציה אסינכרונית.

בתוכה נוצר client אסינכרוני.

ה-client שולח בקשה למודל.

await ממתין לתשובה.

asyncio.run מפעיל את כל המנגנון.

כאשר מבינים את הדוגמה הזו, הרבה יותר קל להבין את הקובץ הבא, שבו המרצה מראה את היתרון האמיתי של async: לא רק קריאה אחת למודל, אלא כמה קריאות שרצות במקביל.

המסר של הפרק הוא ש-async אינו קישוט תחבירי. הוא הכנה לצורת עבודה שמתאימה למערכות AI אמיתיות, שבהן הקריאות למודלים ולשירותים חיצוניים הן חלק משמעותי מזמן הריצה של המערכת.
