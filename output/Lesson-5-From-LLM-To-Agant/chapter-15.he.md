# Error Handling בעבודה עם LLM

אחד הדברים שמפתיעים הרבה מפתחים בתחילת העבודה עם LLMs הוא כמה מהר עוברים מדמו יפה למציאות מורכבת.

בדמו הכול עובד:

- שולחים prompt

- מקבלים תשובה

- מציגים אותה למשתמש

אבל במערכת אמיתית, במיוחד כאשר AI הופך לחלק מ-flow עסקי, העולם נראה אחרת לגמרי.

קריאות נכשלות. 
רשת נופלת. 
המודל מחזיר JSON לא תקין.

Rate limits מופיעים.

Credentials פגי תוקף.

Timeouts קורים.

ולפעמים המודל פשוט מחזיר משהו שלא עומד בכלל במה שביקשנו.

ולכן Error Handling הוא לא "שיפור נחמד".

הוא חלק מהארכיטקטורה.

בלי שכבת הגנה אמיתית, מערכות AI נהיות שבירות מאוד.

כאן מתחיל ההבדל בין:

- דמו 
לבין

- מערכת production

בקובץ anthropic_api_structured.py כבר ראינו התחלה של טיפול נכון בשגיאות.

לדוגמה:

```python
@retry(
    stop=stop_after_attempt(3),
    wait=wait_fixed(2),
    retry=retry_if_exception_type(Exception),
)
```

הקוד משתמש ב-Tenacity כדי לבצע retry אוטומטי כאשר משהו נכשל.

אבל כדי להבין למה זה חשוב, צריך להבין אילו סוגי שגיאות בכלל קיימים בעבודה עם LLMs.



## שגיאות זמניות

אלו השגיאות הנפוצות ביותר.

לדוגמה:

- timeout רגעי

- connection reset

- בעיית DNS

- latency חריגה

- שירות שלא זמין לכמה שניות

- internal server error זמני

בדרך כלל אלו שגיאות שלא באמת דורשות שינוי בקוד.

אם ננסה שוב בעוד שנייה או שתיים, יש סיכוי טוב שהכול יעבוד.

לדוגמה:

```bash
HTTP 500
Connection timeout
Service unavailable
```

במקרים כאלה retry הוא פתרון מצוין.

וזו בדיוק המטרה של Tenacity.

במקום שהמשתמש יקבל שגיאה מיד, המערכת מנסה שוב בצורה אוטומטית.

לדוגמה:

```python
@retry(
    stop=stop_after_attempt(3),
    wait=wait_fixed(2),
)
```

המשמעות:

- עד 3 ניסיונות

- המתנה של 2 שניות בין ניסיונות

אם התקלה הייתה רגעית, המשתמש אפילו לא ירגיש שמשהו נכשל.



## שגיאות throttling ו-Rate Limits

אלו שגיאות נפוצות מאוד בעבודה מול ספקי AI.

לדוגמה:

429 Too Many Requests

המשמעות:

שלחנו יותר מדי בקשות בזמן קצר.

זה קורה הרבה כאשר:

- יש עומס

- עובדים במקביליות גבוהה

- מריצים batch גדול

- משתמשים ב-worker queues

- כמה שירותים חולקים אותו API key

במקרה כזה retry בדרך כלל כן הגיוני, אבל לא בצורה אגרסיבית.

אם נמשיך לשלוח בקשות מיד, רק נחמיר את הבעיה.

לכן נהוג להשתמש ב-Exponential Backoff:

```python
from tenacity import wait_exponential

@retry(
    wait=wait_exponential(
        multiplier=1,
        min=1,
        max=30,
    )
)
```

כל ניסיון מחכה יותר זמן:

- 1 שנייה

- 2

- 4

- 8

- 16

כך נותנים לשירות החיצוני זמן להתאושש.

זו טכניקה בסיסית כמעט בכל מערכת מבוזרת מודרנית.



## שגיאות פורמט

אלו שגיאות מסוג אחר לגמרי.

כאן הקריאה עצמה הצליחה. 
המודל ענה. 
אבל הפלט לא תקין.

לדוגמה:

- JSON שבור

- שדה חסר

- טיפוס לא נכון

- ערך שלא עומד ב-schema

- טקסט חופשי במקום אובייקט

**לדוגמה:**

```python
{
  "participants": "John"
}
```

כאשר ה-schema ציפה ל:

```python
{
  "participants": []
}
```

או מצב שבו המודל בכלל מחזיר:

```bash
The customer sounded upset and requested compensation.
```

במקום JSON.

זו נקודה קריטית מאוד:

אסור לסמוך על המודל רק כי ביקשנו יפה.

LLMs הם מנגנונים הסתברותיים.

הם לא parser קשיח.

גם אם 95% מהפעמים הם מחזירים מבנה נכון, production systems חייבים להתמודד עם 5% האחרים.

ולכן בקוד מופיעה ולידציה:

```python
validate(instance=parsed, schema=schema)
```

זו אחת השורות החשובות ביותר בכל המערכת.

היא אומרת: 
לא מספיק שהמודל החזיר משהו שנראה נכון. 
צריך לבדוק אותו.

וזה שינוי מחשבתי חשוב.

הרבה מתחילים עובדים עם LLM כך:

```python
response = llm(...)
data = json.loads(response)
```

וזהו.

אבל במערכת אמיתית זה מסוכן.

צריך לבדוק:

- האם זה JSON תקין

- האם כל שדות החובה קיימים

- האם הטיפוסים נכונים

- האם המבנה תקין

- האם נוספו שדות לא צפויים

- האם הערכים עומדים בחוקים העסקיים

LLM output חייב לעבור verification.

תמיד.

## שגיאות קריטיות

יש שגיאות שאין טעם לבצע עליהן retry.

לדוגמה:

```bash
401 Unauthorized
Invalid API Key
Missing credentials
```

אם ה-API key לא תקין, retry לא יעזור. גם בעוד שעה זה ייכשל.

אותו דבר לגבי:

- credentials חסרים

- tenant לא נכון

- endpoint שגוי

- הרשאות חסרות

במקרים כאלה צריך לעצור מיד ולהחזיר שגיאה ברורה.

**ופה חשוב להבין:** 
Retry לא נועד "להעלים שגיאות".

Retry מיועד רק לבעיות זמניות.

אחד האנטי-patterns הכי נפוצים הוא לבצע retry על כל Exception.

לדוגמה:

```python
retry_if_exception_type(Exception)
```

זה אולי נוח בדמו, אבל במערכות גדולות זה מסוכן.

למה?

כי זה עלול:

- להעמיס על השירות

- להסתיר באגים אמיתיים

- לגרום לעיכובים מיותרים

- להקשות על debugging

- ליצור cascading failures

במערכות production נהוג להיות הרבה יותר מדויקים.

לדוגמה:

```python
retry_if_exception_type(
    (
        TimeoutError,
        ConnectionError,
    )
)
```

**ופה מגיעה נקודה עמוקה יותר:**

לא כל כישלון הוא אותו סוג של כישלון.

מערכת טובה יודעת להבחין בין:

- transient failure

- permanent failure

- malformed output

- infrastructure issue

- business validation issue

זו בדיוק החשיבה שמתחילה להופיע כאשר AI הופך לחלק ממערכת אמיתית.

## למה חייבים לבדוק את הפלט

אחד הדברים המסוכנים ביותר בעבודה עם LLMs הוא שהפלט לעיתים נראה משכנע גם כשהוא שגוי.

וזה נכון לא רק מבחינת עובדות. גם מבנה יכול להיות "כמעט נכון".

לדוגמה:

```python
{
  "sentiment": "very angry"
}
```

כאשר המערכת ציפתה ל:

```bash
"positive"
"neutral"
"negative"
```

מבחינת בן אדם זה נראה סביר. 
מבחינת מערכת downstream זה עלול לשבור analytics, dashboards או workflows.

לכן AI systems צריכים defensive programming.

הגישה הנכונה היא:

המודל מנסה

- המערכת בודקת

- רק אז משתמשים בתוצאה

לא:

- המודל אמר

- אז בטוח נכון

זה הבדל עצום.
